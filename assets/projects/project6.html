SSAFY 12기 공통 프로젝트로 팀원 6인이 함께 개발한 무인 순찰 로봇 통합 관제 솔루션입니다. ROS2 기반 Gazebo 시뮬레이터와 FastAPI 백엔드, Vue 3 프론트엔드로 구성된 전체 스택을 7주 안에 처음부터 구축한 프로젝트로, 로봇 2대의 실시간 위치 추적·원격 제어·CCTV 스트리밍을 웹 인터페이스에서 통합 관제하는 것을 목표로 했습니다.

프로젝트 종료 시점까지 유휴 상태 CPU 75~80% 점유 문제를 원인 규명 단계에서 마무리하지 못했으며, 이 미해결 과제는 이후 ROBOCOP 로컬 포팅 및 고도화 프로젝트에서 다시 꺼내 해결했습니다.

담당 역할

백엔드 설계 및 구현

FastAPI 기반 REST API 전체 구조를 설계하고, Motor(async MongoDB 드라이버)로 로봇·맵·사용자 컬렉션을 구성했습니다. roslibpy를 활용한 ROS Bridge 클라이언트 연동과 카메라 MJPEG 스트리밍 엔드포인트를 구현했습니다.

로봇 센서 토픽 중개

rosbridge WebSocket을 통해 ROS2 토픽 데이터(/odom, /image_raw, /scan)를 백엔드 서버로 수신하고, SSE 및 MJPEG 스트리밍으로 프론트엔드에 전달하는 파이프라인을 구축했습니다.

배포 환경 설정

Nginx 리버스 프록시와 Docker Compose를 활용해 백엔드·프론트엔드·MongoDB를 컨테이너로 묶고 EC2 인스턴스에 배포했습니다. WSL2 ↔ Windows 호스트 ↔ EC2 간 네트워크 구성과 rosbridge 포트 포워딩을 설정했습니다.

프론트엔드 와이어프레임

Figma로 관제 대시보드 와이어프레임을 설계하고, 팀원과의 역할 분담 기준을 제안했습니다.

미해결 과제 — 유휴 CPU 75~80% 점유

프로젝트 기간 중 유휴 상태 CPU 점유율이 75~80%에 달하는 현상을 발견했지만, 일정 내에 근본 원인을 규명하지 못하고 종료했습니다. 당시 제기한 가설은 다음과 같았습니다.

가설결론
WSL2 리소스 경합Python 프로세스 단독 측정으로 배제 확인
카메라 토픽 대역폭 과부하ROS Bridge 오프라인 상태에서도 동일 현상 → 배제
roslibpy + Twisted busy-wait원인으로 의심했으나 프로젝트 내 제거 미완료

이 미해결 과제는 ROBOCOP 로컬 포팅 및 고도화에서 roslibpy 제거 + websockets/asyncio 전환 + uvicorn --reload 제거를 통해 유휴 CPU 0.3% 수준으로 해결되었습니다.
